✅ Clean C# Solution (Sorting + Two Pointers + Index Preservation)
=================================================================

public class Solution {  
    public int\[\] TwoSum(int\[\] nums, int target) {  
  
        int n \= nums.Length;  
  
        // Step 1: Create (value, index) pairs  
        (int value, int index)\[\] arr \= new (int, int)\[n\];  
        for (int i \= 0; i < n; i++) {  
            arr\[i\] \= (nums\[i\], i);  
        }  
  
        // Step 2: Sort based on value  
        Array.Sort(arr, (a, b) \=> a.value.CompareTo(b.value));  
  
        // Step 3: Two pointer approach  
        int left \= 0, right \= n \- 1;  
  
        while (left < right) {  
            int sum \= arr\[left\].value + arr\[right\].value;  
  
            if (sum \== target) {  
                return new int\[\] { arr\[left\].index, arr\[right\].index };  
            }  
            else if (sum < target) {  
                left++;  
            }  
            else {  
                right\--;  
            }  
        }  
  
        return new int\[0\]; // fallback (won’t happen as per problem)  
    }  
}

Store previous elements for O(1) lookup  
  
\## When to Use  
\- Pair sum problems  
\- Frequency counting  
\- Complement search  
  
\## Example Problems  
\- Two Sum  
\- Subarray Sum = K  
  
\## Template  
for each element:  
    if complement exists:  
        return  
    store element

\# DSA Pattern Learning Toolkit  
  
\## Overview  
  
This project is a personal AI-assisted system designed to learn Data Structures and Algorithms (DSA) through pattern recognition, not just problem solving.  
  
Instead of focusing on finding the fastest solution, the system helps the user explore multiple approaches to the same problem, each revealing a different way of thinking.  
  
The goal is to build a reusable mental toolkit of patterns that can be applied across problems.  
  
\---  
  
\## Problem Statement  
  
Most people approach DSA by solving many problems and memorizing solutions.  
  
This leads to:  
  
\- Forgetting solutions over time  
\- Inability to adapt to new problems  
\- Weak pattern recognition  
\- Over-reliance on memorization instead of understanding  
  
This system addresses that by shifting focus from:  
  
\> solving problems → learning patterns  
  
\---  
  
\## Goal  
  
To help the user:  
  
\- Solve problems using multiple approaches  
\- Understand the reasoning behind each approach  
\- Identify and extract reusable patterns  
\- Build a personal library of patterns over time  
\- Develop problem-solving intuition  
  
\---  
  
\## Key Principles  
  
\### 1. Multiple Approaches per Problem  
  
Each problem should be solved in more than one way when possible.  
  
Example:  
  
\- Brute force → exhaustive thinking  
\- HashMap → memory-based optimization  
\- Two pointers → structure-based reasoning  
  
\---  
  
\### 2. Patterns over Solutions  
  
Solutions are not the end goal.  
  
Each solution should map to a pattern such as:  
  
\- Complement lookup (HashMap)  
\- Two pointers  
\- Prefix sum  
\- Sliding window  
  
\---  
  
\### 3. Guided Learning, Not Auto-Solving  
  
The AI should:  
  
\- Ask for the user’s approach first  
\- Provide hints progressively  
\- Avoid giving the full solution immediately  
\- Encourage thinking and exploration  
  
\---  
  
\### 4. Learning Through Mistakes  
  
Mistakes are treated as learning signals.  
  
The system captures:  
  
\- What went wrong  
\- Why it went wrong  
\- What insight was gained  
  
\---  
  
\### 5. Pattern Reinforcement  
  
Patterns are:  
  
\- Tagged to problems  
\- Stored in a pattern library  
\- Revisited during revision  
  
\---  
  
\## System Components  
  
\### 1. AI Learning Engine  
  
Guides the user through:  
  
\- Understanding the problem  
\- Exploring approaches  
\- Debugging logic  
\- Extracting patterns  
  
\---  
  
\### 2. Problem Repository  
  
Each problem contains:  
  
\- Problem statement  
\- Multiple solution approaches  
\- Notes and reflections  
  
\---  
  
\### 3. Pattern Library

1\. User provides problem

This is not:

This is:

\# Learning Philosophy  
  
\## Overview  
  
This system is built on the idea that solving problems is not the goal.  
  
The goal is to learn how to think.  
  
Most learners struggle with DSA not because they cannot understand solutions, but because they do not build reusable thinking patterns.  
  
This system focuses on developing those patterns through guided exploration.  
  
\---  
  
\## Core Belief  
  
\> Understanding comes from thinking, not from seeing solutions.  
  
\---  
  
\## Principles  
  
\### 1. Struggle is Necessary  
  
Learning requires effort.  
  
The system should not remove difficulty completely. Instead, it should guide the user through it.  
  
\- Productive struggle is encouraged  
\- Frustration is reduced through hints, not answers  
\- The user should arrive at key ideas whenever possible  
  
\---  
  
\### 2. Thinking Before Answering  
  
The user should always attempt the problem first.  
  
The system should:  
  
\- Ask for the user’s approach  
\- Understand their reasoning  
\- Build on their thinking  
  
Not:  
  
\- Provide immediate solutions  
\- Override user exploration  
  
\---  
  
\### 3. Progressive Hinting  
  
Help should be given in levels:  
  
Level 1:  
\- Conceptual nudges  
  
Level 2:  
\- Directional hints  
  
Level 3:  
\- Pattern identification  
  
Level 4:  
\- Near-solution guidance  
  
The system should move gradually between levels based on user need.  
  
\---  
  
\### 4. Multiple Ways of Thinking  
  
Each problem should expose different ways of solving it.  
  
Example:  
  
\- Brute force → exhaustive thinking  
\- HashMap → memory-based thinking  
\- Two pointers → structure-based thinking  
  
The goal is not just optimization, but perspective.  
  
\---  
  
\### 5. Patterns are the Real Knowledge  
  
Solutions are temporary.  
  
Patterns are reusable.  
  
The system should always ask:  
  
\- What pattern is being used?  
\- Where else can this apply?  
  
\---  
  
\### 6. Mistakes are Learning Signals  
  
Errors are valuable.  
  
The system should help the user identify:  
  
\- What went wrong  
\- Why it went wrong  
\- What principle was missed  
  
These should be recorded and revisited.  
  
\---  
  
\### 7. Data Modeling is Key  
  
Many problems are solved by representing data correctly.  
  
The system should emphasize:  
  
\- What to store  
\- Why to store it  
\- How representation affects the solution  
  
\---  
  
\### 8. Reflection Reinforces Learning  
  
After solving a problem, the user should reflect:  
  
\- What did I learn?  
\- What mistake did I make?  
\- What pattern did I discover?  
  
Reflection converts experience into memory.  
  
\---  
  
\## Learning Loop  
  
The system follows a continuous loop:  

  

Attempt → Struggle → Hint → Insight → Solve → Reflect

  

  
\---  
  
\## Role of AI  
  
The AI is not a solver.  
  
The AI is a guide.  
  
It should:  
  
\- Ask questions  
\- Provide hints  
\- Highlight patterns  
\- Encourage thinking  
  
It should not:  
  
\- Jump to answers  
\- Over-explain prematurely  
\- Remove thinking effort  
  
\---  
  
\## What Success Looks Like  
  
The user:  
  
\- Recognizes patterns across problems  
\- Derives solutions independently  
\- Understands tradeoffs between approaches  
\- Feels confident exploring new problems  
  
\---  
  
\## Final Principle  
  
\> One problem, multiple perspectives, one deeper understanding.

\# Pattern System  
  
\## Overview  
  
The pattern system is the core of this learning toolkit.  
  
Instead of treating each problem as unique, this system organizes problems around reusable patterns.  
  
A pattern represents a way of thinking that can be applied across multiple problems.  
  
\---  
  
\## What is a Pattern  
  
A pattern is a reusable problem-solving strategy.  
  
It answers:  
  
\- What idea solves this class of problems?  
\- When should this idea be used?  
\- How can it be applied?  
  
Examples:  
  
\- HashMap (complement lookup)  
\- Two pointers  
\- Prefix sum  
\- Sliding window  
  
\---  
  
\## Why Patterns Matter  
  
Solving problems individually leads to memorization.  
  
Recognizing patterns leads to understanding.  
  
With patterns:  
  
\- New problems feel familiar  
\- Solutions can be derived, not recalled  
\- Thinking becomes structured  
  
\---  
  
\## Pattern Structure  
  
Each pattern should be stored in a structured format:  
  
\### Name  
Example: HashMap (Complement Lookup)  
  
\### Core Idea  
Store previously seen values to enable fast lookup  
  
\### When to Use  
\- Pair sum problems  
\- Complement-based problems  
\- Frequency tracking  
  
\### Recognition Signals  
\- “Find two elements”  
\- “Check if something exists quickly”  
\- Repeated lookups  
  
\### Template (Mental Model)  

  

for each element:  
  
compute needed value  
  
check if needed exists  
  
if yes → return  
  
store current element

  

  
\### Example Problems  
\- Two Sum  
\- Subarray Sum Equals K  
  
\---  
  
\## Pattern Categories  
  
Patterns can be grouped into categories:  
  
\### 1. Lookup Patterns  
\- HashMap  
\- Set  
  
\### 2. Two Pointer Patterns  
\- Opposite ends  
\- Sliding window  
  
\### 3. Accumulation Patterns  
\- Prefix sum  
\- Running totals  
  
\### 4. Structural Patterns  
\- Sorting + logic  
\- Greedy selection  
  
\---  
  
\## Problem to Pattern Mapping  
  
Each problem should map to one or more patterns.  
  
Example:  

  

Two Sum → HashMap, Two Pointers

  

  
Rules:  
  
\- A problem can have multiple patterns  
\- Each pattern should represent a distinct way of thinking  
\- Avoid duplicate or redundant mappings  
  
\---  
  
\## Pattern Extraction Process  
  
After solving a problem, the system should:  
  
1\. Identify approaches used  
2\. Map each approach to a pattern  
3\. Record the mapping  
4\. Update pattern examples  
  
\---  
  
\## Pattern Reinforcement  
  
Patterns should be reinforced through repetition:  
  
\- Same pattern across different problems  
\- Different contexts, same idea  
\- Increasing difficulty  
  
\---  
  
\## Pattern Library Growth  
  
The library grows over time:  

  

Problem → Approach → Pattern → Library Update

  

  
Each new problem:  
  
\- Strengthens existing patterns  
\- Introduces new variations  
\- Improves recognition ability  
  
\---  
  
\## Pattern Index  
  
A central index maps problems to patterns:  
  
\`\`\`json id="ps5"  
{  
  "Two Sum": \["HashMap", "Two Pointers"\],  
  "Subarray Sum": \["Prefix Sum", "HashMap"\]  
}

\# Learning Workflow  
  
\## Overview  
  
This workflow defines how a user interacts with the system while solving a problem.  
  
The goal is to make the process feel natural and conversational while guiding the user through multiple ways of thinking.  
  
The workflow is adaptive and does not require explicit commands.  
  
\---  
  
\## High-Level Flow  

  

Problem → Thinking → Guidance → Solution → Alternative Approaches → Patterns → Reflection

  

  
\---  
  
\## Stage 1: Problem Intake  
  
\### Trigger  
User provides a problem statement.  
  
\### System Behavior  
  
\- Restate the problem in simple terms  
\- Highlight:  
  - input  
  - output  
  - constraints (if relevant)  
  
Then ask:  
  
\- “How would you approach this?”  
  
\---  
  
\## Stage 2: User Thinking  
  
\### Trigger  
User shares their approach  
  
\### System Behavior  
  
\- Validate correct reasoning  
\- Identify gaps or inefficiencies  
\- Ask guiding questions such as:  
  
  - Can we reduce time complexity?  
  - Do we need to check all pairs?  
  - What information would help us avoid repeated work?  
  
Do not provide solutions at this stage.  
  
\---  
  
\## Stage 3: Guided Discovery  
  
\### Trigger  
User is stuck, incomplete, or partially correct  
  
\### System Behavior  
  
Provide hints progressively:  
  
Level 1:  
\- Conceptual nudge  
  
Level 2:  
\- Directional hint  
  
Level 3:  
\- Pattern-level hint  
  
Level 4:  
\- Near-solution guidance  
  
Move to the next level only if needed.  
  
\---  
  
\## Stage 4: First Solution  
  
\### Trigger  
User arrives at a working solution  
  
\### System Behavior  
  
\- Confirm correctness  
\- Discuss time and space complexity  
\- Reinforce the core idea  
  
\---  
  
\## Stage 5: Alternative Approach Exploration  
  
\### Trigger  
First solution is complete  
  
\### System Behavior  
  
Introduce a different way of thinking.  
  
Do not say:  
\- “Here is another solution”  
  
Instead ask:  
  
\- “Can we solve this without extra space?”  
\- “What if the array was sorted?”  
\- “Can we use structure instead of memory?”  
  
Goal:  
Expose a new pattern, not just optimize.  
  
\---  
  
\## Stage 6: Multi-Approach Completion  
  
\### Trigger  
User explores additional approaches  
  
\### System Behavior  
  
Ensure each approach represents a distinct idea:  
  
\- Brute force → exhaustive thinking  
\- HashMap → memory-based thinking  
\- Two pointers → structure-based thinking  
  
Avoid redundant variations.  
  
\---  
  
\## Stage 7: Pattern Extraction  
  
\### Trigger  
Problem is solved using one or more approaches  
  
\### System Behavior  
  
Ask:  
  
\- What pattern did you use?  
\- Why did it work?  
\- When would you use it again?  
  
Map:  

  

Problem → Patterns

  

  
\---  
  
\## Stage 8: Reflection  
  
\### Trigger  
End of problem  
  
\### System Behavior  
  
Prompt the user:  
  
\- What mistake did you make?  
\- What did you learn?  
\- What would you do differently next time?  
  
Encourage concise notes.  
  
\---  
  
\## Stage 9: Persistence  
  
\### Trigger  
After reflection  
  
\### System Behavior  
  
Save:  
  
\- problem  
\- approaches  
\- patterns  
\- notes  
  
Update:  
  
\- pattern index  
\- pattern library  
\- lessons log  
  
\---  
  
\## Adaptive Behavior  
  
The workflow adjusts based on user:  
  
\### Beginner  
\- More guidance  
\- Simpler hints  
\- Fewer abstractions  
  
\### Intermediate  
\- Pattern-focused hints  
\- Faster transitions  
  
\---  
  
\## Key Rules  
  
\- Do not jump to solutions  
\- Do not suggest multiple approaches upfront  
\- Introduce alternatives only after first solution  
\- Always connect solutions to patterns  
\- Keep interaction conversational  
  
\---  
  
\## End Goal  
  
Each problem should result in:  
  
\- at least one working solution  
\- at least one identified pattern  
\- at least one insight or lesson  
  
\---  
  
\## Final Principle  
  
\> A problem is complete only when its pattern is understood.

Bad:

\### 10. No Em Dashes or En Dashes  
  
\- Do not use — or –

\---

\- Validate correctness

Introduce via questions, not statements.  
  
Example:

\## Debugging Behavior

\# HashMap Pattern  
  
\## Core Idea  
  
Store previously seen elements to enable fast lookup in O(1) time.  
  
Used when you need to check if something exists while iterating.  
  
\---  
  
\## When to Use  
  
\- Pair sum problems  
\- Complement-based problems  
\- Frequency counting  
\- Duplicate detection  
  
\---  
  
\## Recognition Signals  
  
\- "Find two elements"  
\- "Check if something exists quickly"  
\- "Avoid nested loops"  
\- Repeated lookups  
  
\---  
  
\## Mental Model  
  
For each element:  
1\. Compute what you need  
2\. Check if it already exists  
3\. If yes → solution found  
4\. Otherwise → store current element  
  
\---  
  
\## Template  

  

for each element:  
  
needed = target - current

  

if needed exists in map:  
    return result  
  
store current in map

  

  
\---  
  
\## Example Problems  
  
\- Two Sum  
\- Subarray Sum Equals K  
\- Contains Duplicate  
  
\---  
  
\## Variations  
  
\- Store value → index  
\- Store value → frequency  
\- Store prefix sum → index  
  
\---  
  
\## Common Mistakes  
  
\- Storing before checking (can reuse same element)  
\- Losing index information  
\- Not handling duplicates correctly  
  
\---  
  
\## Insight  
  
Trade memory for speed.  
  
HashMap removes the need for nested loops.

Markdown

\# Two Pointer Pattern  
  
\## Core Idea  
  
Use two indices to traverse the array from different directions to reduce search space.  
  
\---  
  
\## When to Use  
  
\- Sorted arrays  
\- Pair problems (sum, difference)  
\- Range shrinking problems  
  
\---  
  
\## Recognition Signals  
  
\- "Find two elements"  
\- "Array is sorted"  
\- "Find pair with target sum"  
\- Need to optimize O(n²)  
  
\---  
  
\## Mental Model  
  
Start from both ends:  
  
\- If sum too big → move right pointer left  
\- If sum too small → move left pointer right  
  
\---  
  
\## Template  

  

left = 0  
  
right = n - 1

  

while left < right:  
  
sum = nums\[left\] + nums\[right\]

  

if sum == target:  
    return  
  
elif sum < target:  
    left += 1  
  
else:  
    right -= 1

  

  
\---  
  
\## Example Problems  
  
\- Two Sum (sorted)  
\- 3Sum  
\- Container With Most Water  
  
\---  
  
\## Variations  
  
\- Opposite ends  
\- Sliding window (dynamic window)  
\- Fast and slow pointers  
  
\---  
  
\## Common Mistakes  
  
\- Using on unsorted array  
\- Forgetting to preserve original indices after sorting  
\- Wrong pointer movement logic  
  
\---  
  
\## Insight  
  
Use structure instead of memory.  
  
Reduces time without extra space.

\# Prefix Sum Pattern  
  
\## Core Idea  
  
Precompute cumulative sums to answer range or subarray queries efficiently.  
  
\---  
  
\## When to Use  
  
\- Subarray sum problems  
\- Range sum queries  
\- Repeated sum calculations  
  
\---  
  
\## Recognition Signals  
  
\- "Subarray sum"  
\- "Sum of elements between indices"  
\- Repeated sum computation  
\- Need faster than O(n²)  
  
\---  
  
\## Mental Model  
  
Instead of recalculating sums:  
  
Store running total:  

  

prefix\[i\] = sum of elements from 0 to i

  

  
Then:  

  

sum(i to j) = prefix\[j\] - prefix\[i-1\]

  

  
\---  
  
\## Template  

  

prefix\[0\] = nums\[0\]

  

for i in range(1, n):  
  
prefix\[i\] = prefix\[i-1\] + nums\[i\]

  

  
\---  
  
\## Example Problems  
  
\- Subarray Sum Equals K  
\- Range Sum Query  
\- Continuous Subarray Sum  
  
\---  
  
\## Variations  
  
\- Prefix sum + HashMap  
\- 2D prefix sum  
\- Running sum without array  
  
\---  
  
\## Common Mistakes  
  
\- Off-by-one errors  
\- Incorrect subtraction logic  
\- Forgetting base cases  
  
\---  
  
\## Insight  
  
Avoid recomputation by storing cumulative results.

Store previously seen elements to enable fast lookup in O(1) time.  
  
\---  
  
\## When to Use  
  
\- Pair sum problems  
\- Complement problems  
\- Frequency counting  
\- Duplicate detection  
  
\---

\- "Find two elements"  
\- "Check if something exists"  
\- Repeated searching  
  
\---  
  
\## Template  
  
for each element:  
    needed = target - current  
  
    if needed exists:  
        return  
  
    store current  
  
\---  
  
\## Common Mistakes  
  
\- Storing before checking  
\- Losing index information  
\- Not handling duplicates

Step 3 — Identify the Type of Mistake

📄 Example: `debug-patterns/order-of-operations.md`
===================================================

  

Markdown

\# Debug Pattern: Wrong Order of Operations  
  
\## Mantra  
  
\- Check before storing  
\- Order matters  
  
\---  
  
\## Description  
  
The logic is correct, but steps are executed in the wrong order.  
  
\---  
  
\## Example  
  
HashMap:  
  
Wrong:  
\- store first  
\- then check  
  
Correct:  
\- check first  
\- then store  
  
\---  
  
\## Recognition Signals  
  
\- Using same element twice  
\- Duplicate issues  
\- Unexpected early matches  
  
\---  
  
\## Fix Strategy  
  
\- Re-evaluate step sequence  
\- Ask: what should happen first?  
  
\---  
  
\## Insight  
  
Correct logic in wrong order is still wrong.

  

* * *

  

📄 Example: `debug-patterns/lost-data.md`
=========================================

  

Markdown

\# Debug Pattern: Lost Data  
  
\## Mantra  
  
\- Don’t lose information during transformation  
  
\---  
  
\## Description  
  
Important information is lost when modifying data (sorting, filtering, etc.)  
  
\---  
  
\## Example  
  
Sorting array removes original indices  
  
\---  
  
\## Recognition Signals  
  
\- Correct values but wrong output  
\- Missing original references  
  
\---  
  
\## Fix Strategy  
  
\- Attach metadata (value, index)  
\- Preserve identity  
  
\---  
  
\## Insight  
  
If output needs something, you must carry it along.
